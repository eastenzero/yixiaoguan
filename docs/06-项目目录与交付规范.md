# 项目目录与交付规范

## 1. 目标

本文件用于规定后续项目目录、文档层级、知识条目结构和交付物命名方式，保证后续扩展时不出现结构混乱。

## 2. 当前项目目录结构

当前阶段建议的一级结构如下：

```text
/
├─ docs/
├─ knowledge-base/
├─ apps/
├─ services/
├─ deploy/
└─ scripts/
```

说明：

- 当前已实际建立的是 `docs/` 与 `knowledge-base/`
- 其余目录是后续工程落地时的预留结构

## 3. 文档目录规范

`docs/` 分为两层：

- 正式文档层：`00-07`
- 底稿层：`legacy/`

规则：

- 总册永远保持唯一入口
- 专题文档按职责分工，不相互重复
- 历史底稿不删，但不继续承担主导航职责

## 4. 知识库目录规范

`knowledge-base/` 当前结构：

- `raw/`
- `entries/`
- `templates/`

其中：

- `raw/` 对应原始材料层
- `entries/` 对应知识条目层
- `templates/` 对应条目模板和规范辅助文件

## 5. 知识条目命名规范

统一格式：

```text
YYYYMMDD-分类-主题短名.md
```

例如：

```text
20260323-奖助贷补-国家助学金申请流程.md
```

## 6. 后续工程目录预留原则

前端工程：

- `apps/teacher-web/`
- `apps/student-app/`
- `apps/admin-web/`

后端工程：

- `services/business-api/`
- `services/ai-service/`

辅助目录：

- `deploy/` 用于部署脚本和环境模板
- `scripts/` 用于知识整理、导入、转换等脚本

## 7. 当前交付物定义

当前阶段的正式交付物是文档资产而不是代码资产，包括：

- 项目总册
- 项目定位文档
- 架构与技术路线文档
- 产品模块文档
- 阶段路线文档
- 知识治理文档
- 目录与交付规范文档
- 风险与验收文档
