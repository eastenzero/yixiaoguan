# 格式转换执行方案

## 1. 定版结论

当前知识库源数据的格式转换方案正式定版为：

- `md`：直接保留，统一为 UTF-8
- `docx`：`pandoc -> Markdown`
- `doc`：先转 `docx`，再 `pandoc -> Markdown`
- `xlsx`：按工作表导出 `CSV`
- `xls`：先转 `xlsx`，再按工作表导出 `CSV`
- `pdf`：统一走 `MinerU API`
- `pptx`：暂不批量自动化，文本型优先转 `PDF` 后再走 `MinerU API`
- `jpg/png`：仅在缺少文字版时作为 OCR 候选
- `zip/rar`：先解压，再按内部文件类型处理

统一中间产物为：

- 主文本：`Markdown`
- 表格：`CSV`
- 结构化附加结果：`JSON`
- 原件：保留不动

## 2. 为什么采用这套方案

原因很直接：

- `doc/docx/xls/xlsx` 在本机最稳的方案仍然是 `Office COM + pandoc`
- `pdf` 现在已经通过 `MinerU API` 实测打通，不再依赖手工点桌面版
- 这套组合更适合先把高价值材料转出来，再进入清洗和知识抽取

## 3. 各格式处理规则

### 3.1 Markdown

- 保留原始层级和标题
- 统一编码为 UTF-8
- 不重写原文，只做清洗前预处理

### 3.2 DOCX

- 直接使用 `pandoc` 转成 GitHub Flavored Markdown
- 保留标题、列表、表格、粗体等基础结构

推荐命令：

```powershell
pandoc -f docx -t gfm --wrap=none -o output.md input.docx
```

### 3.3 DOC

- 先用 `Word COM` 转成 `docx`
- 再走 `pandoc`
- 原始 `.doc` 保留，不作为最终文本层

### 3.4 XLSX / XLS

- 以工作表为单位导出 `CSV`
- 表格文件默认不直接作为问答主文本
- 如含规则说明页，仅抽取说明页文本，其余工作表作为附件数据保留

### 3.5 PDF

统一口径：

- 全部优先走 `MinerU API`
- 不再依赖 `MinerU.exe` 手工界面
- 标准产物应包括：
  - 原始下载包 `*.zip`
  - API 元数据 `*.result.json`
  - 解压后的完整结果目录
  - 便于直接抽取的 `full.md`

知识抽取优先使用：

1. 解压目录中的 `full.md`
2. 必要时参考 `content_list_v2.json`、`layout.json`
3. 图片仅用于复核，不作为首选主文本

### 3.6 PPTX

- 当前数量少，暂不优先建设专门流水线
- 文本型 PPTX 可先导出 PDF，再走 MinerU API
- 图示型 PPTX 先保留为附件

### 3.7 JPG / PNG

- 仅作 OCR 候选
- 若已有同内容 Word/PDF 文字版，则图片不单独进主知识流程

### 3.8 ZIP / RAR

- 只视为容器
- 先解压，再按内部文件类型重分类

## 4. 当前推荐工具链

按当前阶段，推荐使用以下分工：

- 盘点与索引：PowerShell / Python 脚本
- `doc/docx`：`Word COM + pandoc`
- `xls/xlsx`：`Excel COM`
- `pdf`：`MinerU API`
- 知识抽取：基于转换结果再交给大模型

## 5. 不再采用的旧口径

以下旧口径不再使用：

- `pdf` 优先走桌面版 MinerU
- 为了 PDF 转换先折腾 `MinerU CLI`
- 把桌面版结果目录作为正式主流程依赖

当前明确以 `MinerU API` 为 PDF 正式方案，CLI 或桌面版仅作为备选排障手段。

## 6. 推荐执行顺序

建议后续按这个顺序推进：

1. 先完成材料筛选和价值分级
2. 批量转换 `doc/docx`
3. 批量导出 `xls/xlsx -> CSV`
4. 批量执行 `pdf -> MinerU API`
5. 在转换结果上补齐元数据、切块和知识条目抽取

## 7. 与后续清洗的关系

格式转换不是终点，而是数据清洗的前置层。

完成转换后，才能稳定进入下一阶段：

- 去重
- 时效判断
- 规则抽取
- 条目拆分
- 标准答复生成
