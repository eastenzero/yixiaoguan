# 材料总表字段规范

## 1. 目标

材料总表是 `数据库部分材料/` 与 `knowledge-base/entries/` 之间的中间索引层，用于：

- 统一盘点原始材料
- 支持后续格式转换
- 为模型抽取提供稳定输入
- 记录分类、时效、版本和处理动作

材料总表应优先采用表格或 `csv` 管理，便于批量处理。

## 2. 一份材料一行

原则上每个原始文件占一行，不按条目拆分。

例外：

- 同一压缩包解包后，内部文件单独建行
- `学生手册.md` 这类超长总册可先作为一行总材料，后续再按章节拆“候选条目表”

## 3. 核心字段

以下字段为首轮必须字段：

| 字段名 | 含义 | 示例 |
| --- | --- | --- |
| `material_id` | 材料唯一编号 | `MAT-20260323-0001` |
| `source_path` | 原始文件相对路径 | `数据库部分材料/数据库部分材料/2025年工作/...` |
| `file_name` | 文件名 | `2025届毕业离校通知.docx` |
| `extension` | 扩展名 | `.docx` |
| `source_dir` | 一级来源目录 | `2025年工作` |
| `sub_dir` | 二级来源目录 | `2025上-21级毕业工作` |
| `title_guess` | 清洗后的暂定标题 | `2025届毕业离校通知` |
| `year_label` | 从标题或目录提取的年份/学年 | `2025` |
| `audience` | 适用对象 | `毕业生` |
| `knowledge_category` | 映射后的知识分类 | `就业与毕业` |
| `material_type` | 材料类型 | `通知` |
| `value_level` | 价值分级 | `A/B/C/D` |
| `timeliness` | 时效状态 | `现行/阶段性/待确认/过期` |
| `is_template` | 是否模板 | `是/否` |
| `is_duplicate` | 是否重复或近重复 | `是/否` |
| `primary_reference` | 主参考材料ID（通常填写主参考 `material_id`） | `MAT-20260323-0005` |
| `processing_action` | 建议处理动作 | `可转知识/仅作附件/待OCR/待解包/暂缓` |
| `notes` | 备注 | `疑似与某通知重复` |

## 4. 推荐扩展字段

以下字段建议在第二轮补齐：

| 字段名 | 含义 |
| --- | --- |
| `effective_date` | 生效日期或通知日期 |
| `updated_date` | 文件落款或更新日期 |
| `issuer` | 发文单位或责任部门 |
| `source_type_detail` | 更细粒度类型，如“通知+附件包”“操作说明”“证明模板” |
| `conversion_status` | 是否已转文本 |
| `conversion_path` | 转换后的文本路径 |
| `model_status` | 是否已进入模型抽取 |
| `entry_candidate_count` | 预估可拆出的知识条目数 |
| `review_status` | 是否已人工复核 |

## 5. 字段取值规范

### 5.1 `knowledge_category`

仅允许以下取值：

- `入学与学籍`
- `奖助贷补`
- `就业与毕业`
- `心理与测评`
- `证件与校园服务`
- `竞赛与第二课堂`
- `事务申请与审批`
- `学院通知与常见流程`
- `待定`

### 5.2 `material_type`

建议统一使用以下枚举：

- `通知`
- `制度`
- `流程说明`
- `操作手册`
- `模板`
- `申请表`
- `汇总表`
- `名单`
- `统计表`
- `截图/图片`
- `压缩包`
- `综合材料`

### 5.3 `processing_action`

建议统一使用以下枚举：

- `可转知识`
- `提炼规则后入库`
- `仅作附件`
- `待OCR`
- `待解包`
- `待确认`
- `暂缓`
- `过期归档`

### 5.4 `primary_reference`

统一填写该材料对应的主参考 `material_id`：

- 独立材料可填写自身 `material_id`
- 重复版本、历史版本、模板平行版本应填写其对应主参考材料ID
- 不再使用 `是/否` 布尔语义

## 6. 建议的工作流字段变更

材料总表在不同阶段会逐步补充：

1. 首轮盘点：填核心字段
2. 清洗复核：修正分类、时效、重复关系
3. 格式转换：补 `conversion_status`、`conversion_path`
4. 模型抽取：补 `entry_candidate_count`、`model_status`
5. 条目发布：关联条目路径或条目编号

## 7. 首批处理范围定义

首批材料建议限定在高价值范围，避免一开始铺太大：

### 范围 A

- `数据库部分材料/学生手册.md`

目的：

- 作为全局规则底稿
- 优先拆出“入学与学籍”“奖助贷补”“心理与测评”“证件与校园服务”“竞赛与第二课堂”类条目候选

### 范围 B

- `数据库部分材料/数据库部分材料/2025年工作/2025下-25级新生工作/`

目的：

- 识别新生高频问题
- 校验学生手册与实际通知是否存在口径差异

### 范围 C

- `数据库部分材料/数据库部分材料/2025年工作/2025上-21级毕业工作/`

目的：

- 建立毕业、离校、就业、档案、征兵相关知识候选

### 范围 D

- `数据库部分材料/数据库部分材料/模板/` 中与奖助贷补、第二课堂、证明模板直接相关的内容

目的：

- 提前识别模板类材料，避免后续误入知识主库

## 8. 建议的首批产出

首批应至少产出三份内容：

1. 完整材料总表
2. 高价值材料子集
3. 候选知识条目清单

其中“候选知识条目清单”建议字段包括：

- `material_id`
- `candidate_question`
- `knowledge_category`
- `audience`
- `priority`
- `source_path`

## 9. 与知识条目的衔接

材料总表不是知识库本身，但后续每个知识条目都应能反向关联到材料总表中的 `material_id` 和 `source_path`。

这样可以解决三个问题：

- 可追溯
- 可复核
- 可定位旧版本和替代材料
