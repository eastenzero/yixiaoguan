# 风险、依赖与验收原则

## 1. 主要风险

需求扩散风险：

- 如果把过多长期能力提前拉入近期建设，项目会重新回到比赛式堆功能。

知识库失真风险：

- 如果未先完成知识治理就直接推进问答，后续会出现知识质量不稳、追溯困难和返工。

对接难度低估风险：

- 官网、统一身份认证、学工和教务等系统对接存在明显外部依赖，不适合排在前期。

运营缺位风险：

- 如果没有持续更新机制，知识库很快会失效，AI 回答质量也会下降。

文档职责混乱风险：

- 如果没有统一总册和专题分工，文档很快会再次重叠、冲突、失去导航作用。

## 2. 当前阶段依赖

项目当前继续推进前，至少依赖以下前置条件：

- 项目总册明确
- 阶段路线明确
- 技术路线明确
- 知识治理规则明确
- 高优先级知识样本可整理

## 3. 各阶段验收原则

一期验收原则：

- 两条主闭环成立
- 学生端、教师端、管理端都有明确落点
- 知识条目具备持续增长入口

二期验收原则：

- 平台能力从单点闭环扩展到多事务支持
- 通知和知识运营能力明显增强

三期验收原则：

- 核心平台能力稳定后，再进入校内系统集成
- 多学院和通用模块抽象具备清晰边界

## 4. 当前文档体系的验收原则

本轮文档重构完成后，应满足：

- 从总册进入，能快速理解全局
- 技术、产品、路线、知识库、规范、风险各有唯一主文档
- 现有底稿文档都有明确归属
- 一期、二期、三期边界清楚
- 后续如需扩写到实现层，可在现有专题文档下继续展开
