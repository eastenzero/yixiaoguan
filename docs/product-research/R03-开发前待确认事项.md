## B 类：基础设施 ✅

### B1：云服务商
**✅ 已确认：腾讯云（域名/备案），阿里云 DashScope（AI API）**

- **两家云混用完全没问题**，AI API 调用是 HTTP 请求，跟服务器在哪家云毫无关系
- 服务器推荐落在**腾讯云**（域名备案在那里，HTTPS 证书申请更顺畅）

### B2：对象存储
**✅ 已确认：云 OSS**

- 推荐：**腾讯云 COS**（与服务器同区域，内网流量更快，账单整合统一）

### B3：数据库
**✅ 已确认：自建 PostgreSQL**（在服务器上安装）

- 一期规模用 Docker 跑 PG 完全足够，自己备份成本低

### B4：部署方式
**✅ 已确认：单台服务器 + 本地开发后推送部署**

**关于"单台服务器能否同时跑所有服务"：**
> 完全可以。推荐用 **Docker Compose** 统一管理：
> - PostgreSQL 容器
> - Redis 容器
> - Spring Boot API 容器
> - FastAPI AI 服务容器
> - Nginx 容器（反向代理 + 静态文件）
>
> 对象存储（COS）在云上，不占服务器资源。本地开发和服务器用同一套 Docker Compose 配置（只换环境变量），推送后直接 `docker compose up -d` 即可。

**关于本地开发流程：**
- 本地装 Docker Desktop，`docker compose up` 即可跑全套
- 代码改好 → git push → 服务器 git pull + docker compose 更新，标准流程

## 专题讨论 1：学生端 H5 vs 小程序

**你的目标**：最终形态是微信小程序，但一期希望先看到效果。

**关键分析**：

| 方案 | 开发成本 | 可见效果 | 升级路径 |
|---|---|---|---|
| 先做 H5（纯网页） | 最低，AI 写页面最快 | 能看，但手机体验一般 | H5 → 小程序需要重写 |
| 直接做小程序 | 需要开发者资质、真机调试 | 接近最终效果 | 已是终态 |
| **uni-app 同时支持两端** | 一套代码，编译两端 | H5 先上，小程序后发布 | ✅ 最佳路径 |

**建议**：**一期用 uni-app 写学生端，先以 H5 形式发布测试，积累完资质和测试后再发布小程序版。**  
代码不重写，路径最优。

## 专题讨论 2：从零写 vs 二改现成框架

**你的问题**：前后端都可以考虑，自己写容易丢失页面。

**结论：强烈推荐使用现成框架二改，尤其是 1 个月上线的压力下。**

### 后端推荐：若依（RuoYi-Vue3）

**若依**是国内最流行的 Spring Boot 开源后台框架，开箱即包含：

| 功能 | 是否已有 |
|---|---|
| 用户管理（增删改查、批量导入） | ✅ |
| 角色与权限（RBAC） | ✅ |
| 菜单权限控制 | ✅ |
| 审计日志 | ✅ |
| 代码生成器 | ✅ |
| 文件上传接口 | ✅ |
| Swagger 文档 | ✅ |
| JWT 认证 | ✅ |

**这意味着**：用若依作为底座，用户、角色、权限、日志这四块完全不用写，可以直接专注于问答/申请/知识库业务逻辑。

> ⚠️ 注意：若依绑定了 MyBatis，与 MyBatis-Plus 兼容，但需要按其约定组织代码。

### 前端推荐：若依自带的 Vue3 前端（教师/管理端）

若依官方维护了配套的 Vue3 + Element Plus 前端，已经包含：
- 登录页、权限路由、菜单、面包屑
- 用户管理、角色管理页面
- 通用表格、表单、弹窗组件

教师工作台和管理端的大多数页面可在此基础上**添加业务页面即可**，不需要搭架子。

### 学生端（uni-app）
- 用 DCloud HBuilderX 的 uni-app 模板初始化
- 找一个成熟的 uni-app Chat 模板二改（开源极多）

### AI 服务（FastAPI）
- 从零搭，因为现有 `kb_vectorize.py` 和 `kb_demo_app.py` 已有基础，封装成 FastAPI 接口即可
- 代码量不大，自己写反而比套框架更干净
