## 一、产品定位

**医小管**定位为面向高校学生事务与辅导员场景的**校园智能服务平台**，目标是从"比赛演示作品"转向"可上线、可交接、可演进"的正式系统。

核心价值主张：
- 成为学生的"24小时虚拟辅导员"，解决高频咨询、事务导办、申请办理等场景
- 成为教师/辅导员的"问题分流与减负工具"，聚合学生问题并提供高效处理工作台
- 成为管理侧的"知识治理与运营平台"，支撑知识持续积累与质量管控

---

## 二、三类用户与核心诉求

| 用户 | 核心需求 |
|---|---|
| 学生 | 问问题、找流程、提申请、查进度 |
| 教师/辅导员 | 看提问、处理提问、审批事项、沉淀知识 |
| 管理员/运营 | 审核知识、维护分类、管理权限、查看日志 |

## 三、一期功能意向（已冻结范围）

### 学生端
- 智能问答（RAG 知识检索 + AI 回答）
- 事务导办（办事流程说明 + 官网链接聚合）
- 空教室申请
- 申请进度查询
- 常用官网入口聚合

### 教师端
- 今日学生提问列表
- 高频问题汇总与归并
- 待人工处理问题处理
- 空教室申请审批
- 人工答复一键转知识库草稿

### 管理端
- 知识条目审核（三级审核：初审→复审→终审）
- 标签与分类管理
- 用户角色管理（RBAC）
- 系统配置
- 审计日志

---

## 四、两条核心业务闭环

**闭环 A：问答与知识增长**
```
学生提问 → AI 回答 → 未解决问题上报 → 教师人工处理
→ 教师标准答复转知识草稿 → 管理员审核发布
→ 后续学生提问命中新知识
```

**闭环 B：事务申请与审批**
```
学生发起空教室申请 → 教师审批 → 状态通知 → 学生查询进度 → 全流程留痕
```

## 五、确定不做（一期显式排除项）

- 原生 Android/iOS App
- 复杂微服务架构
- 统一身份认证正式对接
- 官网/学工/教务系统正式对接
- 心理预警、用户画像、推荐系统、社区内容互动
- 多模态问答
- 大规模独立向量引擎（如 Milvus）

## 六、技术选型（已确定）

### 前端
| 端 | 框架 | 说明 |
|---|---|---|
| 教师端 Web | Vue 3 + TypeScript + Element Plus | 后台工作台，信息密度高 |
| 学生端 | uni-app + TypeScript | 兼顾 H5 与微信小程序 |
| 管理端 | 并入教师端后台体系 | |

### 主业务后端
- Java 17/21 + Spring Boot 3
- Spring Security（认证与权限）
- MyBatis-Plus
- Maven + Lombok + MapStruct

### AI 与知识服务
- Python 3.11+ + FastAPI
- RAG 检索（pgvector 起步）
- 可选：LangChain/LlamaIndex 做 Prompt 编排
- 可选：Celery/RQ 做任务队列

### 数据层
| 组件 | 用途 |
|---|---|
| PostgreSQL | 主业务数据库 |
| Redis | 缓存、会话、热点数据、限流 |
| MinIO / OSS / COS | 原始文件与附件存储 |
| pgvector | 第一阶段向量检索 |

### 部署层
- Docker + Nginx + Linux

### 技术路线一句话总结
> **Java 管业务，Python 管 AI，Vue 管多端。**

## 七、架构原则（已确定）

1. **先单体，后拆分**：第一版不追求微服务，采用"模块化单体 + 独立 AI 服务"
2. **按职责拆分**：主业务系统 / AI 知识服务 / 前端系统 / 知识资产层
3. **AI 必须独立**：AI 层变化频繁，需与主业务后端解耦，防止实验性迭代干扰业务稳定性
4. **知识库是核心资产**：不是附属资料区，而是整个平台的运营核心
5. **教师工作台是一等公民**：不能只是数据看板，必须是真正的"问题处理与审批工作台"

## 八、知识库体系（已确定）

### 双层结构
- **原始材料层**（`raw/`）：保留来源文件，负责来源追溯
- **知识条目层**（`entries/`）：标准化 Markdown 条目，负责检索与问答

### 主分类（8 大域）
- 入学与学籍
- 奖助贷补
- 就业与毕业
- 心理与测评
- 证件与校园服务
- 竞赛与第二课堂
- 事务申请与审批
- 学院通知与常见流程

### 知识治理流程
`采集 → 清洗 → 分类 → 初审 → 复审 → 终审 → 发布 → 生效 → 更新 → 下线/归档`

## 十、三期路线（已确定）

| 期 | 重点 | 进入条件 |
|---|---|---|
| 一期 | 最小业务闭环（问答 + 申请审批 + 知识沉淀） | 总纲明确、知识样本形成 |
| 二期 | 平台能力扩展（通知中心、更多申请场景、知识运营后台） | 一期闭环稳定可用 |
| 三期 | 校内系统对接与平台化（统一身份认证、多学院） | 核心模块稳定 |
| 长期 | 画像、预警、推荐、多模态 | 不纳入近期 |
