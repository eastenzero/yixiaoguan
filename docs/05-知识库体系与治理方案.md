# 知识库体系与治理方案

## 1. 知识库在项目中的角色

知识库不是附属资料区，而是整个项目最值得保留和放大的核心资产之一。

它承担三类职责：

- 为学生问答和事务导办提供可靠知识底座
- 为教师人工答疑提供沉淀入口
- 为平台持续增长提供可运营内容体系

## 2. 双层结构

知识库统一采用双层结构：

原始材料层：

- 保留通知、手册、模板、汇总表、图片、压缩包等原文件
- 负责来源追溯和附件保存

知识条目层：

- 把可问答内容整理成标准条目
- 作为后续检索、问答和后台运营的主数据

## 3. 分类法

当前主分类统一按业务问题域组织：

- 入学与学籍
- 奖助贷补
- 就业与毕业
- 心理与测评
- 证件与校园服务
- 竞赛与第二课堂
- 事务申请与审批
- 学院通知与常见流程

说明：

- 年份、活动批次、原始文件夹名只保留在来源信息中
- 不作为知识主导航

## 4. 条目格式

知识条目统一采用：

- `Markdown`
- `YAML Frontmatter`

核心字段保留：

- `title`
- `category`
- `audience`
- `summary`
- `answer`
- `source_files`
- `source_type`
- `effective_date`
- `updated_at`
- `status`
- `reviewer`
- `tags`

## 5. 知识来源与处理原则

`doc/docx/pdf`：

- 作为主要来源件
- 可拆解为 FAQ、流程、条件、材料说明

`xls/xlsx`：

- 优先视为名单、模板、汇总表
- 只提炼稳定规则，不直接作为问答主知识

`jpg/png`：

- 主要作为图片附件或 OCR 候选
- 不作为首选主知识

`zip/rar`：

- 仅视为材料容器
- 解包后再分类

## 6. 治理流程

建议的知识治理流程：

1. 采集
2. 清洗
3. 分类
4. 审核
5. 发布
6. 生效
7. 更新
8. 下线/归档

审核机制建议沿用三级思路：

- 初审：整理或运营人员
- 复审：业务老师或辅导员
- 终审：管理员或指导老师

## 7. 项目内的知识增长机制

项目必须具备两条知识增长路径：

- 原始材料整理入库
- 教师人工答疑一键转知识草稿

同时保留相似问题归并能力，帮助发现值得沉淀的新知识。
