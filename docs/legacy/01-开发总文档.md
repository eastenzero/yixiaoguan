# 医小管开发总文档

## 1. 项目目标与一期边界

### 1.1 项目定位

医小管后续建设不再以“比赛 Demo 演示”为目标，而是以“可持续演进、可上线、可交接”的校园智能服务系统为目标推进。

本轮从零重构保留两类资产：

- 业务理解：学生事务、辅导员工作流、学院通知场景、问题分流场景。
- 知识资产：学生手册、办事流程、政策通知、模板材料、常见答疑沉淀。

本轮不保留现有原型代码作为正式版基础，不以兼容旧实现为约束。

### 1.2 建设目标

一期系统需要同时满足以下三个目标：

- 学生端能完成“问答、导办、申请、查询”闭环。
- 教师端能完成“提问处理、审批处理、知识沉淀”闭环。
- 管理端能完成“知识审核、权限管理、审计留痕”闭环。

### 1.3 一期范围

一期只做最小价值闭环，不追求大而全。

学生端一期：

- 智能问答
- 事务导办
- 空教室申请
- 申请进度查询
- 常用官网入口聚合

教师端一期：

- 今日学生提问列表
- 高频问题汇总
- 待人工处理问题
- 空教室审批
- 人工答复转知识库

管理端一期：

- 知识条目审核
- 标签与分类管理
- 用户角色管理
- 系统配置
- 审计日志

### 1.4 暂不纳入一期

以下内容原则上不进入第一阶段开发：

- 原生 Android/iOS App
- 复杂微服务治理
- 过早对接统一身份认证、官网、教务、学工等校内系统
- 心理预警、用户画像、推荐系统、社区、多模态扩展
- 以旧版 Node + JSON 存储继续演化为正式版

## 2. 技术选型与取舍理由

### 2.1 总体技术路线

- 教师端 Web：`Vue 3 + TypeScript + Element Plus`
- 学生端：`uni-app + TypeScript`
- 主业务后端：`Java 17/21 + Spring Boot 3`
- AI 与知识服务：`Python 3.11+ + FastAPI`
- 数据层：`PostgreSQL + Redis + 对象存储 + pgvector`
- 部署层：`Docker + Nginx + Linux`

一句话结论：

> Java 管业务，Python 管 AI，Vue 管多端。

### 2.2 为什么这样选

前端：

- 教师端属于典型后台工作台，Vue 3 + Element Plus 成熟稳定。
- 学生端需要兼容 H5 与微信小程序，uni-app 的多端复用更合适。
- 统一 TypeScript，降低长期维护成本。

主业务后端：

- 正式版核心是角色、权限、审批、日志、配置、通知、接口预留，而不是单纯聊天接口。
- Spring Boot 在规范化业务系统、团队交接和高校系统语境下更稳妥。
- 第一版采用模块化单体，控制复杂度。

AI 与知识服务：

- RAG、Embedding、Prompt、模型适配变动频繁，需要与主业务解耦。
- Python 生态更适合文档解析、向量化、检索和实验迭代。
- 独立服务可降低后续替换模型供应商和策略时的成本。

数据层：

- 核心业务数据是关系型数据，应以 PostgreSQL 作为主库。
- Redis 负责缓存、热点数据、会话和限流。
- 对象存储负责保留原始材料、附件和后续上传文件。
- 第一阶段向量检索优先使用 pgvector，后期再视规模升级。

### 2.3 明确不选的路线

- 不以 Express 作为正式版主业务后端。
- 不以 JSON 文件作为正式业务存储。
- 不在第一版引入复杂微服务。
- 不在第一阶段搭建独立 Milvus 集群。

## 3. 系统模块与角色视图

### 3.1 核心角色

- 学生：提问、查找指引、发起申请、查看进度。
- 教师/辅导员：处理学生问题、审批事务、沉淀标准答复。
- 管理员/运营：审核知识、维护分类、维护权限、查看审计日志。

### 3.2 模块划分

用户与权限模块：

- 登录认证
- 用户资料
- RBAC 权限控制
- 审计日志

问答与会话模块：

- 学生提问
- AI 回答
- 会话记录
- 高频问题归并
- 待人工处理上报

知识库模块：

- 知识条目管理
- 标签与分类
- 来源记录
- 版本管理
- 审核发布
- 下线归档

申请审批模块：

- 空教室申请
- 审批流转
- 状态通知
- 操作留痕

导办与入口聚合模块：

- 办事入口
- 办理说明
- 官网链接聚合
- 系统跳转前置说明

教师工作台模块：

- 今日提问
- 高频问题
- 待处理问题
- 待审批事项
- 一键转知识库
- 数据统计

通知与消息模块：

- 站内消息
- 审批通知
- 申请结果通知
- 风险提示

### 3.3 一期真实业务闭环

一期必须打通两条主闭环：

闭环 A：
学生提问 -> AI 回答 -> 未解决上报 -> 教师人工处理 -> 人工答复转知识条目 -> 管理员审核发布 -> 后续问答命中新知识

闭环 B：
学生发起空教室申请 -> 教师审批 -> 状态通知学生 -> 学生查询进度 -> 全流程留痕

## 4. 分阶段开发顺序与每阶段交付物

整体顺序遵循：

`知识库治理规则 -> 信息架构与模块边界 -> 数据对象与流程 -> 技术骨架 -> 一期闭环功能 -> 多端适配与接入预留`

### 阶段 0：需求冻结与知识库整理规范

目标：

- 锁定一期范围与不做项
- 固化知识库分类法、条目模板、命名规则、审核流程
- 形成开发总文档和知识库整理规范

交付物：

- 开发总文档
- 知识库整理规范
- 一期功能清单
- 原始材料分类清单

### 阶段 1：数据模型与业务流程定稿

目标：

- 明确角色、权限、问答、知识条目、审批单、通知、日志等对象
- 定义学生端、教师端、管理端的主要业务流程
- 画出核心状态流与审核流

交付物：

- 领域模型清单
- 核心流程图或状态流说明
- 一期数据库初版设计说明
- API 草案说明

### 阶段 2：基础工程骨架与环境约定

目标：

- 初始化前端、主业务后端、AI 服务三个工程骨架
- 约定目录、分层、编码规范、环境配置和部署方式
- 搭建本地开发与测试所需基础环境

交付物：

- 仓库结构
- 服务初始化骨架
- 环境变量模板
- 本地开发说明

### 阶段 3：一期核心闭环功能开发

目标：

- 完成学生问答、教师工作台、知识审核、空教室申请审批的最小闭环
- 跑通“提问到知识沉淀”和“申请到审批通知”两条主链路

交付物：

- 学生端一期页面与接口
- 教师端一期页面与接口
- 管理端知识审核能力
- 知识条目入库与查询能力

### 阶段 4：体验补全与接入预留

目标：

- 补充 H5/小程序适配
- 完善消息通知、审计日志、官网入口聚合
- 为统一身份认证、校内系统对接预留接口边界

交付物：

- 多端适配说明
- 接口预留文档
- 审计与通知补充能力

## 5. 仓库/服务/文档目录规划

### 5.1 建议的仓库一级结构

```text
/
├─ docs/
├─ knowledge-base/
│  ├─ raw/
│  ├─ entries/
│  └─ templates/
├─ apps/
│  ├─ teacher-web/
│  ├─ student-app/
│  └─ admin-web/         # 可按需要并入 teacher-web
├─ services/
│  ├─ business-api/
│  └─ ai-service/
├─ deploy/
└─ scripts/
```

说明：

- 当前第一步只建设 `docs/` 和 `knowledge-base/`。
- `knowledge-base/raw/` 用于记录原始材料归属，不在此阶段迁移现有原文件。
- `knowledge-base/entries/` 用于后续沉淀标准知识条目。

### 5.2 当前文档阶段的目录规划

`docs/` 存放：

- 开发总文档
- 知识库整理规范
- 一期功能清单
- 原始材料分类清单

`knowledge-base/templates/` 存放：

- 知识条目模板

`knowledge-base/entries/` 按业务问题域预建分类目录：

- 入学与学籍
- 奖助贷补
- 就业与毕业
- 心理与测评
- 证件与校园服务
- 竞赛与第二课堂
- 事务申请与审批
- 学院通知与常见流程

### 5.3 知识库条目格式约定

知识条目统一采用 `Markdown + YAML Frontmatter`。

推荐文件名格式：

```text
YYYYMMDD-分类-主题短名.md
```

例如：

```text
20260323-奖助贷补-国家助学金申请流程.md
```

## 6. 风险、依赖、验收标准

### 6.1 当前阶段主要风险

- 需求边界继续扩散，重新回到“比赛式堆功能”。
- 在没有整理知识底稿前直接开发问答，导致后续返工。
- 把按年份存档的原始材料误当成面向问答的知识组织方式。
- 低估后续校内系统对接难度。
- 缺少知识库维护角色，导致知识上线后无法持续更新。

### 6.2 当前阶段依赖

- 需要先完成知识库分类法与条目模板。
- 需要明确一期对象与流程后，才能进入数据库设计与接口设计。
- 需要先识别原始材料哪些适合转知识、哪些只能作为附件。

### 6.3 当前阶段验收标准

本阶段完成的标志是：

- 有一份明确的一期开发总文档，可直接指导后续排期。
- 有一份知识库整理规范，可直接指导人工整理材料。
- 有一份一期功能清单，清楚说明“做什么、不做什么、依赖什么”。
- 有一份原始材料分类清单，能覆盖当前目录中的主要材料类型。
- 已建立知识库目录骨架和知识条目模板。
