# 医小管本地知识库整理规范

## 1. 目的与原则

本规范用于指导现有原始材料的整理、提炼、命名、审核和后续入库，目标不是简单归档文件，而是构建一套可持续维护的本地知识库底稿。

整理原则：

- 保留原始材料，不破坏来源链路。
- 先人工清洗，再考虑自动抽取。
- 先形成标准知识条目，再进入问答或向量化。
- 按业务问题域组织，不按年份堆目录。
- 所有知识条目都必须能追溯到来源文件。

## 2. 两层结构

### 2.1 原始材料层

原始材料层用于保留来源文件，只承担“证据、追溯、附件”职责，不直接作为问答主数据。

包括但不限于：

- 政策通知
- 办事指南
- 操作手册
- 模板文件
- 汇总表
- 截图与扫描件
- 压缩包

### 2.2 知识条目层

知识条目层用于承载经过提炼、整理和审核后的标准答案，是后续问答、检索、展示和沉淀的主数据层。

知识条目统一采用：

- 文件格式：`Markdown`
- 元数据格式：`YAML Frontmatter`

选择理由：

- 可直接人工阅读和编辑
- 方便版本管理
- 可轻松转换为数据库记录
- 易于挂接来源文件、状态、更新时间和审核信息

## 3. 分类法

知识主分类统一按业务问题域组织，当前先使用以下一级分类：

- 入学与学籍
- 奖助贷补
- 就业与毕业
- 心理与测评
- 证件与校园服务
- 竞赛与第二课堂
- 事务申请与审批
- 学院通知与常见流程

补充规则：

- 原始材料中的年份、活动名称、年级批次，只作为来源元数据，不作为主分类。
- 同一材料涉及多个问题域时，按主要问答场景归类，必要时通过 `tags` 交叉标注。

## 4. 条目模板与元数据字段

每个知识条目至少包含以下字段：

- `title`
- `category`
- `audience`
- `summary`
- `answer`
- `source_files`
- `source_type`
- `effective_date`
- `updated_at`
- `status`
- `reviewer`
- `tags`

字段说明：

- `title`：面向检索和展示的标准标题。
- `category`：一级分类名称。
- `audience`：适用对象，如学生、教师、管理员、新生、毕业生。
- `summary`：一句话摘要。
- `answer`：标准答复正文，可包含流程、条件、材料、注意事项。
- `source_files`：来源文件路径列表。
- `source_type`：来源类型，如通知、手册、模板、截图、汇总表。
- `effective_date`：政策或流程的生效时间，未知可留空。
- `updated_at`：本条目最近整理时间。
- `status`：建议使用 `draft`、`reviewing`、`published`、`archived`。
- `reviewer`：审核人或责任角色。
- `tags`：辅助搜索标签。

## 5. 原始材料处理规则

### 5.1 `doc` / `docx` / `pdf`

处理原则：

- 视为主要来源件。
- 适用于政策通知、办事指南、制度依据、工作安排、操作手册。
- 可以提炼出 FAQ、流程说明、材料清单、办理条件等知识条目。

默认归宿：

- 原始材料层保留文件
- 知识条目层提炼内容

### 5.2 `xls` / `xlsx`

处理原则：

- 优先视为名单、统计表、模板、汇总表、填报附件。
- 一般不直接进入问答主库。
- 仅提炼其中稳定的规则性内容，如申请条件、字段说明、提交要求、时间节点。

默认归宿：

- 原始材料层保留文件
- 如存在明确规则，再转为知识条目

### 5.3 `jpg` / `png`

处理原则：

- 多数是截图、扫描件、操作步骤图、公告图片。
- 不作为首选主知识。
- 可作为 OCR 候选或条目附件。

默认归宿：

- 原始材料层保留文件
- 必要时 OCR 后辅助生成知识条目

### 5.4 `zip` / `rar`

处理原则：

- 只视为归档容器。
- 不直接参与知识问答。
- 解包后根据内部文件类型重新分类。

默认归宿：

- 原始材料层保留压缩包
- 解包内容按实际类型处理

## 6. 条目编写规则

### 6.1 标题规则

标题应直接对应学生会提出的问题，避免纯行政语气。

推荐写法：

- 国家助学金怎么申请
- 新生学籍注册需要做什么
- 心理测评登录入口在哪里
- 空教室申请需要哪些步骤

不推荐写法：

- 关于开展 2025-2026 学年国家助学金评审工作的通知
- 附件 2：学生申请操作说明

### 6.2 正文结构

建议条目正文统一包含：

- 适用对象
- 办理条件
- 办理流程
- 所需材料
- 办理时间或时限
- 注意事项
- 咨询方式

### 6.3 一条一问

一份原始文件可以拆成多个知识条目，但一条知识尽量只回答一个核心问题，避免把整份通知原样塞进一个条目。

## 7. 命名规则

### 7.1 文件命名

统一格式：

```text
YYYYMMDD-分类-主题短名.md
```

例如：

```text
20260323-入学与学籍-新生学籍注册流程.md
20260323-奖助贷补-国家助学金申请条件.md
```

### 7.2 路径规则

条目路径放入对应业务分类目录，例如：

```text
knowledge-base/entries/奖助贷补/20260323-奖助贷补-国家助学金申请条件.md
```

## 8. 审核与状态流

建议沿用现有材料体现出的三级审核思路：

- 初审：运营或整理人员
- 复审：业务老师或辅导员
- 终审：管理员或指导老师

状态流建议：

`draft -> reviewing -> published -> archived`

使用要求：

- `draft`：刚整理完成，尚未审核
- `reviewing`：进入审核阶段
- `published`：允许进入问答和前台展示
- `archived`：过期、失效或被新版本替代
