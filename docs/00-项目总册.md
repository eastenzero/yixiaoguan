# 医小管项目总册

## 1. 项目概述

医小管的正式建设目标，不再是比赛演示型原型，而是一个面向学生、教师、管理人员的校园智能服务系统。

项目的长期价值不在原有代码，而在以下三类资产：

- 医药管理与学生事务场景理解
- 垂直知识库与高频问题沉淀
- 人机协同处理与知识持续增长机制

项目整体路线遵循一个核心原则：

> 保留知识资产与业务理解，从零建设正式版系统。

## 2. 为什么重构

根据 [医小管从零重构版技术选型说明.md](C:/Users/Administrator/Documents/code/yixiaoguan/医小管从零重构版技术选型说明.md)，当前原型更适合作为需求验证成果，不适合作为正式系统继续迭代，原因主要有：

- 现有实现偏轻量试点，技术底座无法支撑正式业务闭环。
- 已验证的是 MVP，不是完整平台。
- 未来目标包含权限、审批、知识运营和校内系统预留，明显超出原型承载能力。
- 最值得保留的是知识资产，而不是旧代码。

## 3. 项目总目标

全项目目标分为三个面向：

- 学生端：问答、导办、申请、进度查询、官网入口聚合、移动端访问。
- 教师端：提问处理、审批处理、人工统一答疑、知识沉淀、工作台统计。
- 管理端：知识审核、分类管理、权限管理、配置管理、日志审计。

最终希望形成一个可持续扩展的校园智能服务平台，而不是单点 AI 问答页。

## 4. 整体技术路线

总体技术路线固定如下：

- 前端：`Vue 3 + TypeScript`
- 学生端：`uni-app`
- 主业务后端：`Spring Boot 3`
- AI 与知识服务：`FastAPI`
- 数据层：`PostgreSQL + Redis + 对象存储 + pgvector`
- 部署层：`Docker + Nginx + Linux`

一句话概括：

> Java 管业务，Python 管 AI，Vue 管多端。

## 5. 项目阶段路线

项目按“三期建设 + 长期预留”推进。

一期：

- 建立最小价值闭环
- 落地学生问答、事务导办、空教室申请、教师工作台、知识审核、基础权限

二期：

- 扩展平台能力
- 增加通知中心、更多事务申请、更多知识来源接入、知识运营能力、多端体验增强

三期：

- 推进校内系统对接与平台化
- 接入统一身份认证、官网和学工/教务系统
- 支持多学院扩展与通用模块抽象

长期预留：

- 画像、预警、推荐、多模态、深化数据看板
