# 阶段路线与开发顺序

## 1. 总体顺序

全项目开发顺序统一为：

`知识治理与范围冻结 -> 业务与模块边界 -> 工程骨架 -> 一期闭环 -> 平台扩展 -> 校内对接与平台化`

## 2. 一期：最小业务闭环

建设重点：

- 学生问答
- 事务导办
- 空教室申请
- 进度查询
- 教师工作台
- 知识审核
- 基础权限

进入条件：

- 项目总纲明确
- 知识库治理规则明确
- 一期范围冻结
- 高优先级知识条目样本形成

交付结果：

- 问答闭环跑通
- 申请审批闭环跑通
- 教师答疑转知识跑通

暂不做：

- 原生 App
- 校内系统正式对接
- 微服务拆分
- 画像、预警、推荐、多模态

## 3. 二期：平台能力扩展

建设重点：

- 通知中心
- 更多事务申请场景
- 更多知识来源接入
- 知识运营后台完善
- 多端体验增强

进入条件：

- 一期闭环稳定可用
- 知识条目增长机制形成
- 教师侧工作台具备可用性

交付结果：

- 平台从单一闭环扩展到多事务支撑
- 知识库从整理阶段进入持续运营阶段

## 4. 三期：校内系统对接与平台化

建设重点：

- 统一身份认证
- 官网和学工/教务系统接口预留转正式接入
- 多学院扩展
- 通用模块抽象

进入条件：

- 平台核心模块稳定
- 权限、日志、配置、通知能力成熟
- 学校接口条件和资源明确

交付结果：

- 平台具备校内集成能力
- 架构从学院场景迈向通用校园服务平台

## 5. 长期预留

长期预留能力仅保留方向，不纳入近期开工：

- 用户画像
- 风险预警
- 个性化推荐
- 多模态问答
- 深化数据看板
