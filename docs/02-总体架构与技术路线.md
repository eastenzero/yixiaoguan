# 总体架构与技术路线

## 1. 架构原则

项目整体架构遵循以下原则：

- 先单体，后拆分
- 按职责拆分，而不是按技术炫技拆分
- 把教师工作台视为核心能力，而不是附属后台
- 让知识库成为系统能力，而不是零散材料堆积区

## 2. 服务边界

主业务系统：

- 承载用户、角色、权限、审批、通知、日志、配置等稳定业务能力
- 采用 `Spring Boot 3`

AI 与知识服务：

- 承载文档解析、检索、RAG、Prompt 编排、相似问题聚类、知识草稿生成
- 采用 `FastAPI`

前端系统：

- 教师端以 Web 为主
- 学生端以 H5 和小程序为主

知识资产层：

- 原始材料层负责来源保留与追溯
- 知识条目层负责标准化沉淀、检索和问答支撑

## 3. 技术栈结论

前端：

- `Vue 3 + TypeScript`
- 教师端配合 `Element Plus`
- 学生端采用 `uni-app`

后端：

- `Java 17/21 + Spring Boot 3`
- `Spring Security`
- `MyBatis-Plus`

AI 与知识：

- `Python 3.11+ + FastAPI`
- 可按需要接入 RAG 编排与任务队列

数据与部署：

- `PostgreSQL`
- `Redis`
- `MinIO / OSS / COS`
- `pgvector`
- `Docker + Nginx + Linux`

## 4. 为什么不继续沿用旧路线

不继续用旧版 `Node + Express + JSON 文件存储` 的原因不是“不能做”，而是它不适合作为正式版底座，尤其无法稳妥承接：

- 多角色权限
- 审批流转
- 数据一致性
- 留痕审计
- 后续系统集成

## 5. 为什么 AI 独立

AI 相关能力变化快，包含：

- 模型供应商切换
- Embedding 变化
- 检索和重排序策略变化
- Prompt 结构迭代

因此 AI 与知识服务必须与主业务后端分离，避免实验性能力扰动正式业务系统。

## 6. 长期演进原则

- 第一阶段使用模块化单体
- 先稳定领域边界，再考虑拆分
- 先把知识和业务闭环跑通，再做系统对接
- 先用 pgvector 起步，再根据规模决定是否升级独立向量引擎
