# 首批本地知识库 MVP 落地清单（第1步）

## 1. 目标与范围

### 1.1 本阶段目标

- 用当前首批已产出的规则与草稿，先搭建一个可运行、可复核、可持续增量的本地知识库 MVP。
- 先跑通“入库 -> 检索 -> 回答 -> 溯源”闭环，再扩容到更多材料。

### 1.2 本阶段范围

- 纳入：`first-batch-rule-extraction-ready.csv` 对应的 22 条可入库规则与对应草稿。
- 暂不纳入：`P0-REX-0014`（待业务确认后再补入 ready 与草稿链路）。
- 原始材料目录 `数据库部分材料/` 只读保留，不移动、不改名、不覆盖。

## 2. 输入物与前置检查

## 2.1 必备输入物（已存在）

- `knowledge-base/raw/first-batch-processing/manifests/first-batch-rule-extraction-ready.csv`

### 2.2 前置检查清单

- [ ] 每个草稿都具备 `material_id`、`rule_task_id`、`source_files`。
- [ ] 无占位符残留（如 TODO/TBD/待补/占位）。
- [ ] `P0-REX-0014` 仍维持 `manual_review`，不误入 ready。

## 3. 本地知识库构建步骤（MVP）

### 3.1 数据准备

- [ ] 以草稿文件为主语料，按“问题概述/标准答复/办理条件/流程/时间节点/注意事项/依据与证据”拼接正文。
- [ ] 为每条记录补充索引元数据：`entry_id`、`category`、`audience`、`material_id`、`rule_task_id`、`status`。
- [ ] 生成标准化中间数据（建议 JSONL/CSV），放在：
  - `knowledge-base/raw/first-batch-processing/converted/`

### 3.2 切分与向量化

- [ ] 切分策略：优先按章节切分；超长段落按语义+长度补切。
- [ ] 每个分片保留回溯信息：`entry_id` + 章节名 + 原文片段。
- [ ] 完成 embedding 并写入本地向量索引（可先用 `pgvector` 或轻量本地向量库）。

### 3.3 检索与问答链路

- [ ] 实现基础检索：TopK 召回 + 证据返回。
- [ ] 问答输出必须携带来源：至少返回 `entry_id`、`material_id`、证据片段。
- [ ] 对“时效敏感条目”默认附加提醒：以最新通知为准。

### 3.4 最小可用接口

- [ ] 提供最小查询接口（CLI 或本地 API 任一即可）。
- [ ] 支持输入用户问题并输出：标准答复 + 依据片段 + 来源标识。
- [ ] 支持基础日志：查询时间、命中条目、是否命中证据。

## 4. 增量接入机制（清洗并行推进）

### 4.1 增量批次规则

- [ ] 每新增一批 `ready` 规则，按批次追加入索引，不全量重建。
- [ ] 新增批次需先通过人工复核，再进入向量入库。
- [ ] 模板类材料仅提炼稳定规则，不直接整包入问答主知识。

### 4.2 版本与回滚

- [ ] 每次入库生成批次号（例如 `BATCH-YYYYMMDD-NN`）。
- [ ] 保留前一版本索引快照，出现质量回退可一键回滚。

## 5. 验收标准（第1步完成定义）

### 5.1 功能验收

- [ ] 可对首批高频问题完成可读回答。
- [ ] 每条回答均可追溯到具体 `entry_id/material_id`。
- [ ] 未命中时有明确兜底文案，不编造结论。

### 5.2 质量验收

- [ ] 抽检 20 个问题：证据可追溯率 >= 95%。
- [ ] 抽检 20 个问题：回答与草稿口径一致率 >= 90%。
- [ ] 时效敏感问题默认带“需以当年最新公告复核”提示。

### 5.3 数据验收

- [ ] 当前 22 条 ready 全部入库。
- [ ] `P0-REX-0014` 未误入生产索引。
